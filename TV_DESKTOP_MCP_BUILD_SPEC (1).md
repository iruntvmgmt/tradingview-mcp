# TradingView Desktop Controller — MCP Server Build Spec (v3.2)

**Changelog v3 → v3.2:** Folded in the curated subset of the Grok and ChatGPT architecture reviews — selector fallback arrays, CDP target enumeration, connection retry/backoff, typed error classes, recon diffing on rerun, mocked controller tests, factory symmetry for the backtest backend, a `verified` flag per capability, and a lightweight `tv_diagnostics()` tool. Deliberately excluded from this revision: confidence-score floats, DOM fingerprint hashing, structured logging schema, full network-preview/bucket-tagging, and automatic TV/Electron/Chromium version detection — these solve problems a multi-tenant product team would have, not problems a single-operator tool has. Add them later only if a real debugging session demands it.

**Audience:** This document is written for an autonomous coding agent executing the build. Follow phases in order. Do not skip Phase 0. Do not write controller/backend code before `recon_findings.json` exists and has been reviewed. Each phase has explicit deliverables and a "Definition of Done" — do not advance to the next phase until DoD is met.

**Owner context:** This extends an existing `tradingview-mcp` Python MCP server that already has DOM-based Pine Script editor read/write tooling (`hasMonacoGlobal: false`, forcing DOM-scroll fallbacks) and a dynamic verification-timeout function (`computeVerificationTimeout(code)`, clamped to 60s). Reuse existing DOM automation utilities from that codebase where possible instead of rewriting them.

---

## 0. Governing principle

TradingView Desktop is an Electron wrapper around the tradingview.com web app — **not** the embeddable Charting Library SDK. Do not assume any of the following exist as callable JS globals until recon proves it:

- `widget.activeChart()`
- `chart.createStudy(...)`
- `chart.getPineScriptBacktestResults()`
- `chart.dataManager()`

Treat every capability as **unknown** until Phase 0 (Recon) classifies it into one of three implementation paths:

| Path | Meaning | Typical use |
|---|---|---|
| **A — JS** | A real internal JS object/method is reachable via CDP `execute_js` | Fast, stable if found |
| **B — Network** | Data flows over XHR/WebSocket frames that can be intercepted via CDP `Network` domain | Good for streaming data (OHLCV) |
| **C — DOM** | No API/network signal exists; must automate clicks/typing/scraping | Expected default for UI-only features (Pine editor, Strategy Tester) |

No controller method may hardcode a path. Every method's implementation is selected at runtime from a persisted config file (`recon_findings.json`) written by the recon phase. See §4 (Backend Strategy Pattern).

## 2. Phase 0 — Recon (mandatory, blocking)

### 2.1 Purpose

Determine, empirically, which of A/B/C applies to each capability below, and capture the concrete details needed to implement that path (selector strings, JS object paths, network endpoint/message shapes).

### 2.2 Capabilities to classify

- `symbol_control` — changing the active chart symbol
- `timeframe_control` — changing chart resolution
- `indicator_apply` — injecting Pine Script and adding to chart (assume **C**, still verify)
- `indicator_remove` — removing an applied study
- `ohlcv_read` — reading candle data for the active chart
- `backtest_run` — triggering the Strategy Tester for an applied strategy
- `backtest_summary` — reading Overview tab metrics (net profit, win rate, profit factor, max drawdown, etc.)
- `backtest_trade_list` — reading the Trades List tab rows
- `backtest_equity_curve` — reading equity curve data (note: rendered canvas is not viable; look for an underlying numeric table/data attribute instead — do not attempt canvas pixel-reading)
- `screenshot` — capturing the chart (assume CDP `Page.captureScreenshot`, always Path A-equivalent; verify only)

### 2.3 Build `core/services/cdp_connection.py` first

```python
class CDPConnectionManager:
    def launch(self, path: str, port: int = 8315) -> None:
        """
        Launch TradingView Desktop with --remote-debugging-port={port}.
        CDP cannot attach retroactively to an instance not started with
        this flag — do not implement a "connect to whatever is running"
        fallback that silently fails; raise ConnectionSetupError instead
        instructing the user to relaunch via the provided script.
        """

    def connect(self, port: int = 8315, retries: int = 5) -> None:
        """
        Attach to an instance already launched via launch().

        Retry with exponential backoff: 250ms, 500ms, 1s, 2s, 4s.
        If all retries are exhausted, raise ConnectionSetupError with the
        message: "TradingView Desktop is not exposing a remote debugging
        port. Please relaunch using: scripts/launch_tv_desktop.*"
        Never retry indefinitely — fail loudly after the fixed sequence.
        """

    def list_targets(self) -> list[dict]:
        """
        Enumerate CDP targets (Electron can expose more than one: main
        renderer, DevTools window, possibly a splash screen). Return
        each target's type/title/url/id.
        """

    def select_main_renderer_target(self) -> str:
        """
        From list_targets(), pick the correct target to attach to.
        Filter out targets with type == 'other'/'background_page' and
        anything with 'devtools://' in its url. Prefer the target whose
        title/url matches the TradingView app window. If more than one
        candidate remains after filtering, raise ConnectionSetupError
        rather than guessing — this must never silently attach to the
        wrong renderer, since every downstream selector would then fail
        for reasons that look unrelated to target selection.
        This logic must live in exactly this one method so future fixes
        (e.g. TV adds a new window type) are localized.
        """

    def execute_js(self, code: str) -> dict:
        """Evaluate JS in the selected renderer target via CDP Runtime.evaluate."""

    async def listen_network(self, filter_fn) -> "AsyncIterator[dict]":
        """Stream Network domain frames (Network.requestWillBeSent,
        Network.webSocketFrameReceived, etc.) matching filter_fn."""

    def disconnect(self) -> None: ...
```

Also produce a small launch helper script (`scripts/launch_tv_desktop.sh` or `.bat`/`.command` depending on OS — detect from the existing repo's conventions) that always launches TradingView Desktop with the remote-debugging flag baked in, so "connect to running instance" is always valid going forward.

**DoD for 2.3:** `CDPConnectionManager` can launch the real app, `connect()` (with retry) succeeds, `select_main_renderer_target()` returns exactly one target when only TradingView Desktop is running, and `execute_js("1+1")` returns `2`. Write a smoke test that does exactly this and requires a human/agent to confirm the app window actually opened. Also test `select_main_renderer_target()` raising `ConnectionSetupError` when `list_targets()` is mocked to return 0 or 2+ ambiguous candidates.

### 2.4 Build `core/services/recon.py`

```python
class ReconRunner:
    def dump_window_globals(self) -> list[str]:
        """Return window keys filtered for candidates matching
        /trading|chart|study|widget|store|data/i (case-insensitive)."""

    def probe_known_paths(self) -> dict:
        """Try a fixed list of plausible internal globals, e.g.
        window.TradingViewApi, window.__INITIAL_STATE__, window.tvWidget,
        React fiber root detection via document.querySelector('#root')._reactRootContainer,
        etc. Return {path: found_bool, sample_repr: str|None} per candidate."""

    async def tap_network(self, duration_s: int = 30) -> list[dict]:
        """Log all XHR/WS frames for duration_s while a human manually
        interacts with the app (change symbol, timeframe, run backtest).
        Print a countdown/instruction to stdout so the human knows when
        to perform which action, and tag captured frames with a rough
        timestamp bucket so they can be correlated back to the action
        performed at that moment."""

    def dump_dom_structure(self, selector: str) -> str:
        """Snapshot outerHTML (truncated/pretty-printed) for a given
        selector — used against Pine Editor container, Strategy Tester
        panel, and its tab contents."""

    def report(self) -> dict:
        """Assemble all findings into the recon_findings.json shape
        defined in §2.5. This is the ONLY function that writes the file."""
```

**Execution protocol (the agent must follow this interactively, not fully unattended):**

1. Launch app via `cdp_connection.launch()`.
2. Run `dump_window_globals()` and `probe_known_paths()` immediately after load — no user interaction needed for this part.
3. Start `tap_network(duration_s=120)` and print explicit sequenced instructions to stdout, e.g.:
   ```
   [0:00-0:20] Please change the chart symbol to any different symbol now.
   [0:20-0:40] Please change the timeframe now.
   [0:40-1:00] Please open Pine Editor, paste any indicator, click Add to Chart.
   [1:00-1:30] Please open Strategy Tester and let a backtest run to completion.
   [1:30-2:00] Please open the Trades List tab and scroll through it.
   ```
   This is a case where the agent cannot fully self-drive the app (no strategy is loaded yet, no prior UI state) — it must pause and ask the human operator to perform these actions in sequence, then resume automated capture.
4. Run `dump_dom_structure()` against: Pine Editor container, Strategy Tester panel root, Overview tab, Trades List tab, Properties tab. Use whatever top-level selectors are discoverable via `document.querySelector` trial (the agent should try common patterns like `[class*="tv-strategy-tester"]`, `[class*="pine-editor"]`, `[data-name="..."]` and record what actually matched).
5. Call `report()` and write `recon_findings.json`.

```json

    },

    },
    "backtest_summary": {

      "verified": true,
      "detail": { "tab_selectors": ["..."], "row_selectors": {"net_profit": ["..."], "win_rate": ["..."]} }
    },
    "backtest_trade_list": {

    },

      "path": "dom",
      "verified": false,
      "detail": { "fallback": "numeric_table_if_present_else_null" }
    },
    "screenshot": { "path": "cdp", "verified": true, "detail": {} }
  },
  "window_globals_found": ["..."],
  "raw_dom_snapshots": { "pine_editor": "...", "strategy_tester": "..." }

```

Notes on the two additions vs. earlier drafts:

- **Selector arrays, not single strings.** TradingView's CSS class names churn often. `DomUtils` (§3) must try each entry in a `selectors` list in order and use the first one that resolves, rather than failing outright the moment one class name changes. This is the single highest-leverage change in this revision — it directly reduces how often a full recon rerun is needed after a minor TV frontend update.
- **`verified` boolean per capability.** `false` means recon found a plausible candidate but it wasn't exercised end-to-end during the interactive tap (e.g. the human operator skipped that step, or it was inferred rather than observed). Controllers must refuse to construct a backend for an unverified capability unless the operator explicitly passes `allow_unverified=True` — better to fail loudly at startup than to silently automate against a guess. (A finer-grained confidence score was considered and deliberately left out — for a single-operator tool, "trustworthy" vs. "not yet proven" is the only distinction that would actually change behavior.)

**DoD for Phase 0:** `recon_findings.json` exists with `schema_version` set, every key in `capabilities` has a non-null `path` and at least one entry in its selector array (where applicable), and a human has reviewed the file and confirmed the selectors/details look sane (spot-check by re-running `dump_dom_structure` against 2-3 selectors and visually confirming). Do not proceed to Phase 1 without this confirmation.

**On rerun (maintenance runs, not first run):** `tv_recon_run()` must diff the newly generated file against the previously committed `recon_findings.json` and print a short summary before overwriting — e.g. "3 selectors changed in `symbol_control`; `backtest_equity_curve.verified` flipped false→true; `tv_desktop_version` changed." This turns recon into the actual diagnostic tool described in §8, rather than a black box that just replaces the file. Implementation is a plain key-by-key dict diff — no need for DOM fingerprint hashing, since git already retains full history of the committed JSON for anything deeper.

## 3. Phase 1 — `errors.py` and `dom_utils.py`

### 3.0 `core/services/errors.py`

Define typed exceptions up front so every later phase raises the correct one instead of a bare `Exception`. This matters specifically because an agent — not a human — will often be the one catching and reacting to these at runtime; a typed error gives it something to branch on.

```python
class TvMcpError(Exception):
    """Base class for all errors raised by this server."""

class ConnectionSetupError(TvMcpError):
    """CDP launch/connect/target-selection failed."""

class ReconRequired(TvMcpError):
    """recon_findings.json is missing, has an unsupported schema_version,
    or is otherwise unusable. Message must tell the operator to run
    tv_recon_run()."""

class BackendConfigurationError(TvMcpError):
    """recon_findings.json exists but a specific capability's entry is
    malformed (missing path, empty selector array, unsupported path
    value, etc.), or the capability is unverified and
    allow_unverified was not set."""

class CapabilityUnavailable(TvMcpError):
    """Recon explicitly determined this capability has no working
    path (e.g. equity curve has no underlying table)."""

class ElementNotFound(TvMcpError):
    """All selectors in a fallback array failed to resolve within
    timeout. Include the full selector list and timeout in the
    message."""

class SelectorVerificationFailed(TvMcpError):
    """A selector resolved to an element, but a sanity check on its
    content/structure failed (e.g. expected a table, got empty div) —
    signals the UI likely changed shape, not just class names."""

class BacktestTimeout(TvMcpError):
    """wait_for_complete() exceeded its timeout."""

class NetworkCaptureError(TvMcpError):
    """listen_network() failed to establish or lost the CDP Network
    domain subscription."""
```

**DoD:** Every subsequent phase (`dom_utils.py`, backends, controllers) imports from this module and never raises a bare `Exception` or generic `RuntimeError` for the conditions listed above.

## 3.1 `dom_utils.py`

Extract, don't rewrite. Pull the reusable primitives out of the existing Pine editor DOM-scrolling fallback code already in the repo:

- selector-with-retry (wait up to N seconds for element to appear)
- scroll-into-view helper
- hidden/disabled element detection
- text extraction with whitespace normalization
- the existing `computeVerificationTimeout(code)` logic, generalized if needed for non-Pine-editor DOM waits

```python
class DomUtils:
    async def resolve_selector(self, selectors: list[str], timeout_s: float = 10) -> str:
        """Try each selector in `selectors` in order; return the first
        one that resolves to a visible element within timeout_s. Raise
        ElementNotFound (with the full list attached) if none resolve.
        Every other method below should accept `selectors: list[str]`
        and call this internally, rather than taking a single selector
        string — this is what makes the fallback-array design in
        recon_findings.json actually work end to end."""

    async def wait_for_selector(self, selectors: list[str], timeout_s: float = 10) -> bool: ...
    async def scroll_into_view(self, selectors: list[str]) -> None: ...
    async def click(self, selectors: list[str]) -> None: ...
    async def type_text(self, selectors: list[str], text: str, clear_first: bool = True) -> None: ...
    async def extract_text(self, selectors: list[str]) -> str: ...
    async def extract_table(self, table_selectors: list[str], row_selectors: list[str]) -> list[dict]: ...
    async def get_attribute(self, selectors: list[str], attribute: str) -> str | None: ...
    async def is_visible(self, selectors: list[str]) -> bool: ...
    async def hover(self, selectors: list[str]) -> None: ...
    async def focus(self, selectors: list[str]) -> None: ...
    async def wait_until(self, predicate_js: str, timeout_s: float = 10) -> bool:
        """Poll a JS boolean expression via cdp.execute_js until true or
        timeout — for waits that aren't selector-shaped (e.g. 'loading
        spinner class removed AND result count > 0')."""
    async def scroll_paginated_list(self, container_selectors: list[str], row_selectors: list[str], max_scrolls: int = 50) -> list[str]:
        """Handles the same DOM-scrolling fallback pattern used for Pine
        script reads — reuse that logic here for Trades List scraping."""
```

**DoD:** Unit tests against a static local HTML fixture (agent should generate a minimal fixture file mimicking a table + scrollable list) pass for each method, including a case where the first selector in an array fails but the second succeeds (proves fallback actually works, not just that a single correct selector works).

## 4. Phase 2 — Backend Strategy Pattern

This replaces per-method `if path == "dom": ... elif path == "network": ...` branching inside controllers. Controllers call `self.backend.<method>()`; the correct backend class is selected once, at controller construction time, from `recon_findings.json`.

### 4.1 `backends/base.py`

```python
from abc import ABC, abstractmethod

class ChartBackend(ABC):
    @abstractmethod
    async def set_symbol(self, symbol: str) -> None: ...
    @abstractmethod
    async def set_timeframe(self, tf: str) -> None: ...
    @abstractmethod
    async def get_ohlcv(self, limit: int) -> list[dict]: ...
    @abstractmethod
    async def health_check(self) -> bool:
        """Cheap check that this backend's selectors/JS paths still
        resolve, without performing the actual action (e.g. resolve
        the symbol-search selector and confirm it's visible, don't
        actually change the symbol). Used by tv_diagnostics() and
        callable on demand — not run automatically on every server
        startup, since that adds latency to every launch for a check
        you only actually want when something's already broken."""

class IndicatorBackend(ABC):
    @abstractmethod
    async def add(self, pine_code: str, name: str) -> None: ...
    @abstractmethod
    async def remove(self, name: str) -> None: ...
    @abstractmethod
    async def health_check(self) -> bool: ...

class BacktestBackend(ABC):
    @abstractmethod
    async def run(self, strategy_name: str) -> None: ...
    @abstractmethod
    async def get_summary(self) -> dict: ...
    @abstractmethod
    async def get_trade_list(self) -> list[dict]: ...
    @abstractmethod
    async def get_equity_curve(self) -> list[dict] | None: ...
    @abstractmethod
    async def health_check(self) -> bool: ...
```

### 4.2 `backends/js_backend.py`, `network_backend.py`, `dom_backend.py`

Each implements the relevant interface(s) using `cdp_connection.execute_js`, `cdp_connection.listen_network`, or `dom_utils` respectively. Each backend class takes the relevant `detail` dict from `recon_findings.json` in its constructor — no hardcoded selectors inside backend classes; they're parameterized.

```python
class DomChartBackend(ChartBackend):
    def __init__(self, cdp: CDPConnectionManager, dom: DomUtils, detail: dict):
        self.cdp = cdp
        self.dom = dom
        self.detail = detail  # e.g. {"selectors": ["...", "..."]}

    async def set_symbol(self, symbol: str) -> None:
        await self.dom.click(self.detail["selectors"])
        # ... etc, using self.detail's selector arrays, never a literal string

    async def health_check(self) -> bool:
        return await self.dom.is_visible(self.detail["selectors"])
```

**Backend factories** (`backends/__init__.py`) — kept symmetric across all three backend families so the pattern is uniform even where DOM is the only implementation currently in use:

```python
def _get_capability(recon: dict, capability: str) -> dict:
    entry = recon["capabilities"].get(capability)
    if entry is None:
        raise BackendConfigurationError(f"'{capability}' missing from recon_findings.json")
    if not entry.get("verified", False):
        raise BackendConfigurationError(
            f"'{capability}' is unverified — rerun tv_recon_run() or pass allow_unverified=True"
        )
    return entry

def build_chart_backend(capability: str, recon: dict, cdp, dom, allow_unverified: bool = False) -> ChartBackend:
    entry = _get_capability(recon, capability) if not allow_unverified else recon["capabilities"][capability]
    path, detail = entry["path"], entry["detail"]
    if path == "js": return JsChartBackend(cdp, detail)
    if path == "network": return NetworkChartBackend(cdp, detail)
    if path == "dom": return DomChartBackend(cdp, dom, detail)
    raise BackendConfigurationError(f"Unsupported path '{path}' for capability '{capability}'")

def build_indicator_backend(recon: dict, cdp, dom, allow_unverified: bool = False) -> IndicatorBackend:
    entry = _get_capability(recon, "indicator_apply") if not allow_unverified else recon["capabilities"]["indicator_apply"]
    path, detail = entry["path"], entry["detail"]
    if path == "dom": return DomIndicatorBackend(cdp, dom, detail)
    raise BackendConfigurationError(f"Unsupported path '{path}' for indicator_apply")

def build_backtest_backend(recon: dict, cdp, dom, allow_unverified: bool = False) -> BacktestBackend:
    entry = _get_capability(recon, "backtest_summary") if not allow_unverified else recon["capabilities"]["backtest_summary"]
    path, detail = entry["path"], entry["detail"]
    if path == "dom": return DomBacktestBackend(cdp, dom, detail)
    raise BackendConfigurationError(f"Unsupported path '{path}' for backtest capabilities")
```

**DoD:** `test_backends.py` instantiates each backend type against a mock/fixture `detail` dict and confirms method calls dispatch without raising, plus confirms `BackendConfigurationError` is raised for an unverified capability and for an unsupported `path` value. No controller-level tests yet — that's Phase 3.

## 5. Phase 3 — Controllers

### 5.1 `chart_controller.py`

```python
class TVChartController:
    def __init__(self, cdp: CDPConnectionManager, recon: dict, allow_unverified: bool = False):
        dom = DomUtils(cdp)
        self._symbol_backend = build_chart_backend("symbol_control", recon, cdp, dom, allow_unverified)
        self._timeframe_backend = build_chart_backend("timeframe_control", recon, cdp, dom, allow_unverified)
        self._ohlcv_backend = build_chart_backend("ohlcv_read", recon, cdp, dom, allow_unverified)
        self._indicator_backend = build_indicator_backend(recon, cdp, dom, allow_unverified)

    async def set_symbol(self, symbol: str) -> None:
        await self._symbol_backend.set_symbol(symbol)

    async def set_timeframe(self, tf: str) -> None:
        await self._timeframe_backend.set_timeframe(tf)

    async def add_indicator(self, pine_code: str, name: str) -> None:
        """Delegates to self._indicator_backend.add(), which for the DOM
        path should reuse existing Pine editor open/paste/Add-to-Chart
        automation from the current repo's Pine script write path — do
        not duplicate that logic inside the backend, call into it."""
        await self._indicator_backend.add(pine_code, name)

    async def remove_indicator(self, name: str) -> None:
        await self._indicator_backend.remove(name)

    async def get_ohlcv(self, limit: int = 500) -> list[dict]:
        return await self._ohlcv_backend.get_ohlcv(limit)

    async def screenshot(self) -> bytes:
        """Always CDP Page.captureScreenshot — no backend indirection needed."""

    async def health_check(self) -> dict[str, bool]:
        return {
            "symbol_control": await self._symbol_backend.health_check(),
            "timeframe_control": await self._timeframe_backend.health_check(),
            "ohlcv_read": await self._ohlcv_backend.health_check(),
            "indicator_apply": await self._indicator_backend.health_check(),
        }
```

Note the controller never branches on `path` — that decision already happened once, in the `build_*_backend` factories, at construction time.

### 5.2 `backtest_controller.py`

Same pattern, backed by `BacktestBackend`. Default assumption per §0 is DOM for all four capabilities unless recon found otherwise.

```python
class TVBacktestController:
    def __init__(self, cdp: CDPConnectionManager, recon: dict, allow_unverified: bool = False):
        dom = DomUtils(cdp)
        self._backend = build_backtest_backend(recon, cdp, dom, allow_unverified)  # single backend, since all 4 backtest capabilities are almost certainly the same path

    async def run_strategy(self, name: str) -> None:
        await self._backend.run(name)

    async def wait_for_complete(self, timeout_s: float = 60) -> None:
        """Poll DOM for loading indicator gone / results panel populated,
        via dom.wait_until(). Use computeVerificationTimeout-style dynamic
        scaling if backtest duration correlates with bar count / date
        range — otherwise fixed timeout. Raise BacktestTimeout (not a
        bare TimeoutError) if exceeded, so callers can catch it
        specifically."""

    async def get_performance_summary(self) -> dict:
        return await self._backend.get_summary()

    async def get_trade_list(self) -> list[dict]:
        return await self._backend.get_trade_list()

    async def get_equity_curve(self) -> list[dict] | None:
        return await self._backend.get_equity_curve()

    async def health_check(self) -> bool:
        return await self._backend.health_check()
```

## 7. Build order (execute sequentially; each numbered step is a commit-sized unit)

1. `pyproject.toml` — add `playwright`, add `[project.optional-dependencies] recon = [...]` if recon needs extra deps beyond the base server.
2. `core/services/cdp_connection.py` (including `list_targets`/`select_main_renderer_target`/retry-backoff) + smoke test (§2.3 DoD).

4. `core/services/errors.py` — typed exceptions, imported by everything from this point forward.
5. `core/services/dom_utils.py`, extracted from existing Pine editor code, with selector-array fallback support (`resolve_selector`) + fixture tests, including the fallback-actually-works test case.
6. `core/services/backends/base.py` (with `health_check` on every interface), then `js_backend.py`, `network_backend.py`, `dom_backend.py`, then the three factory functions in `backends/__init__.py`, + `test_backends.py` (including the unverified-capability and unsupported-path error cases).

8. `core/services/backtest_controller.py` + `tests/test_backtest_controller.py` (mocked backends).
9. Wire all tools into `server.py`, including `tv_recon_run` and `tv_diagnostics`.
10. End-to-end integration test per §5.2 DoD, run against the real live app on the actual SMC strategy workflow (symbol → timeframe → apply script → backtest → summary → trades → screenshot).

## 8. Maintenance protocol (post-launch)

When TradingView Desktop updates and something breaks:

1. Run `tv_diagnostics()` first — if a `health_check()` failure points at one specific capability, you may not even need a full rerun; a single selector added to that capability's fallback array might fix it.
2. If that's not enough, run `tv_recon_run()` again. It automatically prints a summary diff against the previously committed `recon_findings.json` (e.g. "3 selectors changed in `symbol_control`; `backtest_equity_curve.verified` flipped false→true; `tv_desktop_version` changed") before overwriting.
3. The diff shows exactly which `capabilities[*].detail` selectors changed — only the affected backend class needs a fix, not the controller layer.
4. Commit the updated `recon_findings.json` alongside the backend fix so the two stay in sync.

This is the payoff of the backend/recon separation: breakage is localized and diagnosable instead of requiring a blind selector hunt across the whole codebase.
