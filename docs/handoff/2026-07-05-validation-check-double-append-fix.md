# Handoff: 2026-07-05 — validation_check double-append bug fix

**Agent:** Sage (backend)

**Branch / commit at session end:** `main`

## Bug: run_validation_check appended TWO events per call

`run_validation_check()` in `experiment_controller.py` had a placeholder
append pattern: it wrote a "throwaway" event with
`consecutive_passes_after_this: 0`, then called
`consecutive_validation_passes()` (which counted the placeholder), then
wrote the "real" event.  Result: every validation check produced two
permanent entries in `logs/experiment_log.jsonl`.

After 3 genuine passes, `consecutive_validation_passes()` returned 6, not 3.
Since that count gates `run_holdout_check` via
`validation_passes_required_before_holdout` (default 3), the holdout gate
could be passed roughly twice as early as intended — weakening the exact
overfitting safeguard the controller exists to enforce.

## Fix

- Removed the placeholder `append_event` call entirely.
- Compute `previous_consecutive` from the log BEFORE appending anything.
- If verdict == "pass", consecutive = previous + 1; if "fail", consecutive = 0.
- Append exactly ONE `validation_check` event per call.

## Regression tests added (3)

| Test | What it verifies |
|---|---|
| `test_exactly_one_event_appended_per_check` | `run_validation_check` adds exactly 1 event to log per call |

| `test_pass_pass_fail_pass_returns_one` | pass/pass/fail/pass → 1 (fail resets streak), controller-level parallel of log test |

- **Conclusion**: No production data was corrupted by this bug. The
  only data generated was by test runs, which wrote to temp paths.
